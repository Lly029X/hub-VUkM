### 使用场景：

### 作业

**作业1**（400字文档）: 阅读上面的客服工作台的说明，总结我们自己作为后端开发和算法开发的角色，我们需要做什么？
    ◦ 需要设计数据库吗？
    ◦ 需要使用什么模型？
    ◦ 如何使用使用bert的？
    ◦ 是否需要使用大模型？

1. **需要设计数据库**
   Mysql 作用：存储元数据（文本内容，类目树结构，生效时间，视角，标签，关联问题id等）
   - Category Table (类⽬表): 存储 id , parent_id , name , level 等，⽤于实现⽆限级分类。
   - FAQ Main Table (主表): 存储标准问、答案、答案类型、有效期、视⻆等。
   - Similar Questions Table (相似问表): ⼀对多关联主表，存储那 200 条相似问。
   - Tag/Perspective Mapping (关联表): 存储标签和视⻆的关联关系。
     Milvus 的作⽤： 仅存储 Primary Key （对应 MySQL 的 ID）和 Vector （向量特征）。

2. **需要使用什么模型**
   1. embedding model 将知识库中的问题进行向量化，便于后续作为相似度匹配的基础
   2. Rerank model 在Milvus返回检索到的top 10高相似度问题后，通过BGE-Reranker精排，选出得分最高的
3. **如何使用Bert**
   在FAQ中，Bert不是用来生成回答，而是用来向量化
   1. 预处理：将标准问题和相似问题进行清洗和分词
   2. 向量提取：将文本作为预训练好的BERT模型的输入，取[CLS]位的输出或者mean pooling层的输出作为该问题的向量化表示
   3. 入库：将向量存入Milvus
   4. 检索：用户提问时，通过Bert将用户问题转化为向量，与Milvus中的标准问题进行相似度匹配
4. **是否需要使用大模型？**
   1. 不使用：标准的语义匹配FAQ，根据用户提问的向量在Milvus中召回相似的问题，直接回答事先设置好的答案。
      优点：快速且受控； 缺点：死板，无法灵活应对个人特殊问题
   2. 使用：LLM可以润色用户的口语化表达；在FAQ没有高相似度命中时，可以检索较为相关的FAQ输入给LLM，生成合理的答案。
   3. 意图识别：判断用户是在咨询，闲聊，投诉等
5. **后端算法分工**
   后端：设计数据库，封装Milvus的操作（增删查改向量），逻辑控制（生效时间，视角筛选）
   算法：选取并部署合适的embedding模型服务，优化检索精度（停用词处理，长短句匹配），建立离线/增量索引更新机制

**作业2**（400字文档， 流程图）: 如何使用bert 进行文本编码，并且使用bert 进行相似度计算，需要写清楚技术方案；

1. **分词** **(Tokenization)****：** 使⽤ BERT 的 Tokenizer 将输⼊的 FAQ 标题或相似问切分为 Token，并添加特殊占位符 [CLS] 和 [SEP] 。
2. **特征提取：** 将输⼊喂⼊ BERT 编码器，得到每⼀层 Token 的 Embedding并进⾏池化。
3. **归⼀化** **(Normalization)****：** 对输出向量进⾏归⼀化。

在⽣产环境中，相似度计算分为两个阶段：**离线索引**和**在线检索**。

- 离线索引：

1. **处理相似问：** ⽤户录⼊的 1 条标准问及其 200 条相似问，每⼀条都通过 BERT 转化为⼀个向量。
2. **存储：** * **MySQL****：** 存储 ID 与⽂本内容的映射。**Milvus****：** 存储 (ID, Vector) 。

- 在线索引：

1. **向量化：** ⽤户输⼊新问题 ，调⽤ BERT 服务得到向量 
2. **向量检索** **(Milvus)****：** 使⽤ Search 接⼝，在向量数据库中进⾏检索。
