#   数据库设计
需要设计一个多分类的表结构
在设计表结构时要做好分类
如
```
ID | 标题 | 状态 | 修改人 | 编辑时间 | 生效状态 | 关联问题 | 标签引用 | 答案视角 | 答案类型
```
包含标准问题和 多个问题及相似问题 关联问题 及图片存储
如

```
{
    “from”:"user",
    "value":"我收到的衣服和我下单的衣服颜色不一样",
    "value":"[images]"
}
{
    "from":"boot"
    "value":"亲 抱歉是我们的工作失误呢 这就联系专人帮您处理"
}
```
用户询问了问题还附带了一张图片 这张图片是否需要储存及后续处理

```
{
    "from":"user",
    "value":"这个机箱的尺寸能装的下三风扇的显卡吗 散热限高多少 能上 SFX 电源吗"
}
{
    "from":"boot",
    "value":"亲 这款机箱尺寸为 460.2mm*310.3mm*476.5mm 机箱材质为 SPCC&SECC ABS 塑胶 钢化玻璃,显卡限长 427mm，散热限高 176mm,最高支持 360 冷排及 13 个风扇位，电源最支持 ≤180mm，主板支持 ATX/M-ATX/Mini-ITX 背插主板，且机箱前置配置了 USB3.2 Gen 1 Type-A*2/USB3.2 Gen 2 Type-C*1/耳机&麦克风复合音频接口*1"
```

用户提出了多个问题及相似问题 需要先分出问题类别再一一做出处理

#   模型设计
可以使用

```
正则表达式 + TD-IDF + BERT + LLM
如果条件允许再增加一个 OCR
```
这个组合可以适用于各种应用场景
先用正则筛选如果没有匹配文本再用 TF-IDF 进行特征提取，如没有相似再用 BERT 去精确的分词匹配，最后如果 BERT 无法再数据库中查询到相对应的问题再由 LLM 进行兜底
#   BERT 调试
我们需要对 BERT 进行一个客服场景进行一个微调 比如 这个问题可能会涉及到专业领域的专业术语和参数问题 我们需要拿标准问题和相似问题对他进行一个调整训练
#   是否需要使用大模型
需要
我们在日常使用时可能会涉及到一些比较难懂或者是数据库内没有的问题 这时我们需要大模型来兜底及及时呼叫人工来进行处理
